## Sweep: LFM 0.3.0, Astro 6.3.1, and OpenPanel layered across the fleet

## Why Care?

Three things converged today.

**One**, `@lossless-group/lfm` v0.3.0 went live on JSR â€” the first 0.3.x cut, carrying everything the LFM development branch had been accumulating since 0.2.2 (callouts with brand-source detection, the new `llm-response` and `excerpt` callout types, callout CSS as a two-tier semantic-token surface). For sites consuming LFM as a published dependency rather than a workspace link, this is the upgrade that lets them stop pretending the local `link:` was production-equivalent.

**Two**, Astro 6.3.1 is out and we needed it everywhere. Every actively-maintained site in this pseudomonorepo (and the splash sites that hang off it and its siblings) was lagging â€” most on 6.0â€“6.2, the LFM splash all the way back on Astro 5.16. We took the opportunity to also pick up the minor bumps that had piled up: `@astrojs/vercel`, `@astrojs/check`, `tailwindcss` + `@tailwindcss/vite`, `vitest`, `jsdom`, `typescript`, `@types/node`. Nothing dramatic on its own. All boring, all green.

**Three**, OpenPanel.dev joined the analytics evaluation â€” added _alongside_ the existing Umami and Fathom rather than replacing either. Same `<Analytics />` component pattern in every site, gated on `import.meta.env.PROD && OPENPANEL_CLIENT_ID` so the new block stays inert until each site's clientId is provisioned. We're still comparing platforms; layering preserves the comparison window.

The point isn't that any of these three is hard in isolation. It's that the same per-site `<Analytics />` discipline, the same `pnpm build` verification motion, and the same `link:` vs JSR mode-switch script all held up across eight separate repos with eight separate git histories. The pseudomonorepo posture earned its keep today.

### OpenPanel layered into every Analytics component

- Each site keeps its existing Umami and/or Fathom snippets â€” nothing removed
- Each site's `src/components/Analytics.astro` now reads `OPENPANEL_CLIENT_ID` from `import.meta.env` and emits the OpenPanel init/script pair when both `PROD` and the env var are present
- `lfm/splash` had no Pass-1 analytics at all; it gets OpenPanel as its first tracker
- Three GitHub Pages workflows (`astro-knots/.github/workflows/pages.yml`, `content-farm/.github/workflows/pages.yml`, `lfm/.github/workflows/pages.yml`) were updated to inject `OPENPANEL_CLIENT_ID` from a repo Variable at build time:
  ```yaml
  - name: Build
    run: pnpm build
    env:
      OPENPANEL_CLIENT_ID: ${{ vars.OPENPANEL_CLIENT_ID }}
  ```
  Variable, not Secret â€” `clientId` is publicly visible in rendered HTML by design.
- `.env.example` files updated where present; `lfm/splash` got a new `.env.example` since it had none

The driving prompt â€” `context-v/prompts/Setup-Analytics-Across-Sites.md` â€” was updated with the rollout-status table and the GitHub Pages env-var setup recipe so the discipline is documented, not just performed.

## How the day actually went

The arc was three nested loops.

**Outer loop:** bring `mpstaton-site`'s long-running `development` branch (which had accumulated callouts, wikilink audits, the YouTube playlist family, OG enrichment, analytics experiments) into `main` cleanly, with the LFM dep flipped from local link mode back to JSR for production. The `lfm-mode.mjs` script (which auto-detects branch and swaps `package.json` between `link:../../../lfm` and `npm:@jsr/lossless-group__lfm@<version>`) earned its design â€” a single `pnpm lfm:jsr` call did the package.json edit and the lockfile regen. Build was green on the first attempt.

**Middle loop:** the realization mid-merge that "we might as well update Astro and the rest while we're in here." `pnpm outdated` in each site, sorted by site, showed a consistent shape â€” Astro 6.0â€“6.2 â†’ 6.3.1, plus a clutch of minor patch-bumps across Tailwind, Vercel, vitest, jsdom. Two sites errored on `pnpm outdated` (`Cannot read properties of undefined (reading 'optionalDependencies')` â€” pnpm 10.15 bug, not ours). We worked around by reading the package.json directly and cross-referencing against the npm registry. Eight `pnpm install --ignore-workspace --no-frozen-lockfile` runs in parallel via background tasks, eight `pnpm build` runs to verify. All green.

**Inner loop:** the OpenPanel layer. The original Pass-1 prompt (2026-05-06) had instrumented eight sites with Umami + Fathom in a unified `<Analytics />` pattern. Today's job was to extend the same component, not replace it â€” keep the existing trackers, gate the new OpenPanel block on its own env var, document the per-platform setup in a single-source-of-truth prompt. The pseudomonorepo's instinct toward _conventional shape over abstraction_ paid out here: every site's `Analytics.astro` is a different specific implementation, but they all answer to the same shape, so the diff was identical-modulo-IDs across all eight files.

The one wrinkle worth recording: **GitHub Pages does not read `.env` files at build time**. We had local `.env` files with `OPENPANEL_CLIENT_ID` in each splash, but those don't ship to the build runner. The fix is the workflow `env:` block above, sourcing from a repo Variable. Documented in the prompt so this doesn't get re-discovered next time.
