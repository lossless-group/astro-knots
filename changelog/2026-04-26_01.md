## Changes by Area

### 1. New spec: `context-v/specs/An-Internet-Friendly-Responsive-UI-for-Longform-Writing.md`

A v0.0.1.0 spec (~700 lines) for the long-form reader UI that will publish the Venture Handbook (formerly the in-joke "Voodoo Handbook") and any future books on the FullStack VC site. Status: Proposed.

Major sections:

- **Context** — overview, inspiration (Stripe Press, Tufte, The Pudding, mdBook, Lossless Reader prior art), preferred stack, audience + scale, the **"wrapper stays out of the way" thesis** (load-bearing decision: all content-level features live in `@lossless-group/lfm`, the reader UI handles only sequencing + reading affordances + theme cohesion).
- **Responsive design** — four modes (Mobile / Tablet / Half-Screen / Desktop), triggered by **container width AND aspect ratio** (not viewport-px alone), implemented via container queries. The Half-Screen mode exists specifically because split-screen on a high-DPI laptop is wide-but-portrait — width-only thresholds miss it.
- **Section-Specific & Quote Sharing** — the highest-leverage feature in the spec. Mobile DM has displaced email as the high-trust distribution channel; people share *the part* of a piece that hit them, not the whole chapter. Section + quote shares with OG metadata override, the URL-fragment-doesn't-work problem explained (`#anchor` is never sent to the server, so unfurlers see chapter-level meta), recommended `/share/[book]/[chapter]?h=…&q=…` SSR endpoint approach to preserve CDN caching, W3C Text Fragments interop.
- **Centralized path management** — `src/config/routes.ts` as single source of truth for URL builders AND disk-organization → URL mapping. Disk can nest deeply (`src/content/library/books/venture-handbook/`); URLs stay short (`/read/venture-handbook/strategy-development/`).
- **Component inventory** — 10 components (4 new, several copy-pattern from `packages/lfm-astro/`), homed in `src/components/library/`.
- **Schema additions** — concrete `defineCollection()` block for `src/content.config.ts`.
- **Acceptance criteria** sequenced as v0.1 → v0.2 → v0.3: single chapter render → multi-chapter sequencing + section share → mobile polish + quote share + save-my-place.
- **Yak shaving** — LFM features the reader depends on, in build order: `remark-heading-slugs` first, then `remark-image-directive`, then `remark-toc-extract`, then (stretch) `remark-wikilinks`.
- **10 open questions**, **9 future-plan items**.

Notable design calls captured *with rationale* so they survive future re-reads:

1. **Wing it with LFM** (rejected Starlight, mdBook, Quartz) — same brand chrome as the rest of the site, forcing function for LFM wishlist features that other sibling sites also need.
2. **Container queries over viewport media queries** — driven by user observation that split-screen on his MacBook Pro reports 1732×2158 (wide-but-portrait). Viewport-px alone calls it "Desktop" and ships the full triptych, which then crowds the layout.
3. **`/read/[book]/[chapter]/`** (rejected `/library/`, `/dojo/library/`, `/books/`) — short, verb-clear, SEO-friendly. Disk org stays free to nest deeply.
4. **Section/quote shares > whole-chapter shares** — mobile message sharing has displaced email; URLs need to land on the section, not the chapter top. URL fragments alone don't work for OG (servers never see `#anchor`); we use `?h=` and `?q=` query params combined with `#:~:text=` for native browser highlighting.
5. **`/share/...` SSR endpoint over SSR-everywhere** — preserves CDN caching for the bare chapter URL (the 99% of traffic that's just reading), accepts a small two-route maintenance burden in exchange.

## On "context vigilance" as a practice

The principle behind this entry: every architectural decision worth making is worth writing down with **why**, not just **what**. The spec we shipped today contains:

- ~10 close-call design decisions, each with rationale
- ~10 open questions surfaced explicitly (so they're not surprises six months from now)
- ~9 future-plan items (so the v0.1 implementer doesn't accidentally over-build)
- Concrete code samples for the load-bearing primitives (routes.ts, schema additions, share-URL builder)
- Clear acceptance criteria per version (v0.1 / v0.2 / v0.3)

**Cost**: roughly two hours of focused writing. **Payoff**: a fresh session reading the spec + CLAUDE.md can implement v0.1 without re-litigating any of the design decisions. That payoff compounds every time the spec is re-read — by a future session, by a new collaborator, by an AI assistant who's never seen the codebase, by Mike six months later.

We don't always need this much documentation. We do for surface-area-large work that touches data shape, URL scheme, shared components, or features other sites will want to copy. The Long-Form Reader hits all four.
