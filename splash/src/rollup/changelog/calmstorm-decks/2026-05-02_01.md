## Why Care?

Yesterday's `/thesis` page proved that the **scroll-deck pattern** outperforms the
16:9 letterbox letter-by-letter — typography breathes, sections compose against
real tokens, and the build is shippable on first pass. But it was a single deck
sitting at one URL, with no way to compare alternate layout choices and no chrome
explaining that the page existed at all if you landed at the root.

Today's work answers two questions that emerged the moment a real human looked at
the result:

1. **"Can I see it composed differently without you re-doing it?"** → Yes. There's
   now a second full-deck composition at `/thesis/version-2` with substantially
   different layout choices per slide (17 alternate sections in
   `src/layouts/sections/teaser-v2/`). Same content, different visual language.
2. **"How would a stakeholder ever find any of this?"** → A navigation chrome that
   treats Scroll Decks as a first-class surface alongside the slide-by-slide
   variant chooser, with a toggle that appears in the header only when a scroll
   deck is loaded.

The strategic shift is worth naming: **the slide-by-slide variant chooser remains
the design-research surface (granular A/B per single slide), and the full-deck
scroll variants become the client-facing surface (end-to-end compositions to
share).** Both are now discoverable from the same chrome. Adding a third scroll
deck is one line in a registry file.

## What Was Built

### 1. Scroll Deck — Variant 2 (`/thesis/version-2`)

A second full-deck composition rendered through the same `PageAsDeckWrapper`,
sharing the same source data (`src/data/venture-team.ts`,
`src/data/identity-pillars.ts`) and the same theme tokens. The 17 sections live
in `src/layouts/sections/teaser-v2/` so v1 and v2 evolve independently.

The constraint was: each slide must make a **substantially different layout
choice** from the v1 baseline. Not a re-skin, not a re-color — different
information architecture.

composition diversify cheaply.

### 2. Navigation Chrome — promoting Scroll Decks to first-class

#### `src/lib/scroll-decks.ts` — the variant registry

Single source of truth for "what scroll-deck variants exist." Mirrors the
existing `src/lib/slides.ts` pattern. Currently lists `/thesis` (v1 baseline) and
`/thesis/version-2` (v2 alternate). Exports:

- `SCROLL_DECKS` — ordered array of `{ href, label, variantNumber }`
- `SCROLL_DECK_HOME` — the canonical landing for the header "Scroll" link
- `isScrollDeckPath(pathname)` — used by `DeckHeader` to decide whether to
  render the variant toggle
- `getScrollDeckCycle(pathname)` — returns wrap-around `prev` / `next` plus
  `index` / `total` for any current scroll-deck path

Adding a third scroll deck is one line in this file. The header pill,
`DeckNav` cycling, and the index-page CTA all read from this registry, so
nothing else has to change.

#### `src/components/basics/DeckHeader.astro` — refactored from `SiteHeader`

The thin top header was previously a single component at
`src/components/SiteHeader.astro`. Per astro-knots convention (headers and
footers live under `components/basics/`), it moved via `git mv` to
`src/components/basics/DeckHeader.astro`. Symbol renamed `SiteHeader` →
`DeckHeader` in all 8 importers.

The center column also changed. Previously: a single "Table of Contents" link.
Now: a small nav with **two** surface entries — `Table of Contents · Scroll` —
with the active surface bolded (`text-slate-900 font-medium`) and the inactive
one muted (`text-gray-500`). The decision is path-driven: if
`isScrollDeckPath(Astro.url.pathname)` is true, "Scroll" is active; otherwise
"Table of Contents" is active.

When (and only when) the user is on a scroll-deck page, the header also
renders an **inline variant toggle pill**:

```
Table of Contents · Scroll  〔 ‹  v1 · baseline  (1/2)  › 〕
```

The pill's `‹` and `›` are anchors to the wrapped prev/next variant from
`getScrollDeckCycle()`. Counter scales as more scroll decks are registered.

#### `src/components/basics/DeckNav.astro` — extracted from `SlideLayout`

The fixed bottom-right counter + ‹ › nav, plus the ← / → keyboard handler,
used to be inline inside `SlideLayout.astro`. Pulled out into a presentational
primitive that takes `counter`, `prev`, `next`, `cycling`, optional aria
labels, and an `enableKeyboard` flag (default true). Used by:

- `SlideLayout.astro` — for slide-by-slide variant pages (cycling within a
  slide's variant siblings, or non-cycling across the 17 slides)
- `src/pages/thesis/index.astro` and `version-2.astro` — for cycling between
  scroll-deck variants

This means scroll-deck pages now have **three** discoverable nav affordances
that all do the same thing:

1. The header pill toggle (`‹ … ›`)
2. The bottom-right `DeckNav` (`‹ counter ›`)
3. ← / → keyboard shortcuts

…and they share zero behavior with `PageAsDeckWrapper`'s ↑ / ↓ within-deck
scrolling, so there's no key conflict.

#### `src/layouts/PageAsDeckWrapper.astro` — height becomes overridable

One-line change. The wrapper used to hard-code `height: 100vh`, which broke
when a chrome header (`DeckHeader`, 3rem tall) sat above it — the last slide
was pushed off-screen. Changed to:

```css
.deck-wrapper { height: var(--deck-height, 100vh); }
```

Default behavior is unchanged for any page that uses the wrapper standalone.
The thesis pages set `.thesis-body { --deck-height: calc(100vh - 3rem); }` so
the deck fills exactly the viewport minus the header. No nested scrollbars,
no cutoff.

#### `src/pages/index.astro` — discovery for the new surface

The index page intro was a single column. Restructured into a flex container:

- **Left column (`flex-1`):** the existing `Fund III · Teaser Deck` eyebrow,
  `17 slides` headline, and description. Unchanged copy.
- **Right column (`w-80`):** a slate-tinted aside titled
  "▶ Or scroll the assembled deck" with two stacked cards linking to
  `/thesis` and `/thesis/version-2`. Each card shows the variant label
  ("v1 · the baseline composition", "v2 · 17 alternate layouts") and a `→`
  affordance.
- Below the `lg` breakpoint the columns stack vertically.

Bumped main from `max-w-4xl` → `max-w-6xl` to fit both columns without
cramping the existing copy. The slide list below is untouched.

## What Changed in Approach (the meta-lesson)

Yesterday's lesson was: **AI composition works much better when the model is
composing against pre-decided vocabulary and pre-written content, rather than
inventing both alongside the layout.**

Today extends that with a corollary about **diversification cost**:

> Once the foundation is in place (narratives + tokens + section component
> conventions), generating a second full-deck composition with 17 substantially
> different layouts cost ~one Claude session and required zero changes to data,
> tokens, or the wrapper. The marginal cost of a new full-deck variant has
> collapsed.

This is what makes "promoting Scroll Decks to first-class" tractable as a
product surface. If each new scroll deck cost a week of layout invention, you'd
ship one. Because they cost a session, you can ship many — and the navigation
chrome built today is exactly the affordance that lets clients navigate among
them.

The deeper meta-lesson: **the menu of layout choices is going to expand, and
the chrome should anticipate that.** The registry-driven approach
(`scroll-decks.ts`) means a third variant is one line of TypeScript and zero
template changes. The header toggle, the bottom-right `DeckNav`, the
index-page CTA, and the keyboard handler all read from the same registry.

## Open Items

- **Index page CTA still hardcodes both deck links.** It currently lists the two
  scroll decks in the aside by hand. Should be refactored to map over
  `SCROLL_DECKS` from the registry so adding a third deck propagates here too.
  Small follow-up.
- **No "Scroll Decks" index page yet.** Currently the header "Scroll" link goes
  straight to `/thesis` (v1). When there are 4+ scroll decks it'll likely be
  worth a `/scroll` index that lists them with thumbnails. Deferred until the
  third variant exists.
- **Variant labels live in two places.** The `label` field in `scroll-decks.ts`
  shows in chrome, but the per-page `<DeckHeader meta="…" />` value also
  describes the variant ("Fund III · Teaser · v2"). These should be unified —
  probably `meta` should default to derived-from-registry. Easy follow-up.
- **No content delta between v1 and v2.** Both decks render the same source
  data verbatim. When/if a client wants a v2 with revised copy, that's a
  non-trivial decision (do we fork narrative files? add a copy-overlay system?)
  — flagged for later.
- **The `details>summary` disclaimer in v2's slide 1 is open by default in
  print** because `<details>` doesn't respect `media="print"` natively. Fine
  for now; revisit when print is in scope.
